## 1. 整体架构分层

```

## 3. 模块依赖关系

```
用户服务（User Service）
    ↓
情绪日记服务（Emotion Service） ← 依赖用户服务
    ↓
情绪检测服务（Detection Service） ← 依赖情绪日记 + 用户服务
    ↓
任务服务（Task Service） ← 依赖情绪检测 + 用户服务
    ↓
多周期分析服务（Analysis Service） ← 依赖情绪日记 + 任务服务
    ↓
知识沉淀服务（Knowledge Service） ← 依赖用户服务
    ↓
AI服务（AI Service） ← 依赖所有模块数据（为Karpathy LLM微调做准备）
```

依赖清晰，每个阶段只需要开发依赖已完成的模块，和开发顺序对齐。

## 5. 加密设计

### 5.1 端侧加密流程

因为用户要求云端存储+客户端加密，密钥用户掌握：

1. **用户注册时**：客户端生成公私钥对 / 加密密钥
2. **密钥存储**：
   - 方案A：用户本地保存（安全性最高，但丢失无法找回）
   - 方案B：服务器加密存储密钥（用用户密码加密，可找回）
3. **写入数据时**：客户端对敏感内容加密后再上传云端
4. **读取数据时**：云端返回密文，客户端解密后展示

> 细节留待后续确认，架构上预留加密层即可

### 5.2 加密位置
- 在**数据访问层**之前做加密/解密
- 数据库只存储密文

## 6. 微信接入设计

> **说明**：第一版不开发，**后续迭代**功能。架构预留设计如下：

### 6.1 消息处理流程

```
用户发送消息 → 微信服务器 → 我们的服务回调 → 
  ↓
AI识别内容类型 → 判断是哪个模块
  ↓

  - `/想法 想到一个新点子...` → 直接记录想法

## 11. 关键设计决策总结

| 决策点 | 结论 |
|--------|------|
| 数据隔离 | 所有业务表带 `user_id`，逻辑隔离，不提前分库 |
| 用户体系 | 独立模块，管理后台邀请制，不开放注册 |
| 登录方式 | 微信+账号密码+验证码三剑客（微信登录第一版也支持，只是消息录入入口延后） |
| 加密 | 客户端加密，云端存密文，方案后续细化 |
| 录入入口 | 第一版：Web表单录入；后续迭代：微信统一录入入口，自动识别+快捷命令双模式 |
| 部署 | Docker化，初期单节点Docker Compose足够 |
