# Phase 1: 基础设施 + 用户体系 + 情绪日记核心录入

## 提案概述

本变更实现项目第一阶段开发，完成基础设施搭建、用户体系认证、以及情绪日记模块的核心录入功能。

## 目标

- [x] 完成数据库schema设计（用户表 + 情绪记录表 + 情绪检测结果表）
- [x] 完成后端用户认证API（登录 + 用户管理）
- [x] 完成后端情绪日记CRUD API
- [x] 完成前端响应式布局（PC侧边栏 + 移动端底部导航）
- [x] 完成前端页面：仪表盘概览、情绪记录列表、情绪记录录入/编辑、情绪健康检测、情绪分析回顾、管理员用户管理
- [x] 实现AI自动情绪识别标注功能
- [x] 实现每日情绪健康检测答题功能

## 非目标（后续阶段实现）

- 客户端加密（后续迭代）
- 定时任务主动推送（等微信接入后实现）
- MCP知识库接入（只预留接口）
- 任务列表模块（Phase 3）
- 知识沉淀模块（Phase 5）
- Karpathy LLM微调探索（P1）
- 微信接入（后续迭代）

## 背景

根据产品需求文档，第一阶段需要完成基础设施搭建和核心情绪录入功能。本次变更遵循技术架构文档设计。

## 范围

### In Scope

- 数据库：用户表、情绪记录表、情绪检测结果表、问题池表
- 后端：认证中间件、用户管理API、情绪日记API、情绪检测API、AI服务层入口
- 前端：响应式布局（PC侧边栏 + 移动端底部导航）、仪表盘、情绪列表、情绪录入、情绪检测、情绪回顾、用户管理
- UI/UX：全站响应式支持PC/移动端

### Out of Scope

- 任务列表模块
- 知识沉淀模块
- 客户端加密（架构预留位置，不实现具体加密）
- 定时任务主动推送
- 微信接入
- MCP知识库具体集成

## 风险评估

| 风险 | 影响 | 应对措施 |
|------|------|----------|
| 响应式布局兼容性 | 低 | 遵循Tailwind响应式规范，测试主流断点 |
| AI调用成本 | 中 | 只在用户主动点击时触发识别，不做实时调用 |
| 需求理解偏差 | 低 | 所有关键设计决策已提前探索确认 |

## 设计决策记录

### 1. 客户端加密
**决策**: 开发阶段不实现加密，后续迭代再加加密层
**原因**: 优先保证功能可用，加密不影响架构设计，可以后加

### 2. 定时任务触发
**决策**: 先PULL后PUSH，第一阶段只在用户访问时检查提醒
**原因**: 第一阶段没有微信推送渠道，主动推送意义不大

### 3. AI调用方式
**决策**: 统一后端AI服务层，所有AI调用走后端
**原因**: 保持架构一致性，统一管理API Key

### 4. MCP接入
**决策**: 只预留接口，具体集成后续再做
**原因**: 具体协议和接入信息尚未确定

### 5. 管理员后台
**决策**: 完整实现Web管理后台
**原因**: 邀请制需要管理员能方便地增删用户

### 6. 导航结构
**决策**: PC侧边栏 + 移动端底部图标导航
**原因**: 刚好5个核心模块，移动端底部导航体验更好

### 7. 情绪检测答题
**决策**: 一题一页
**原因**: 用户体验更专注，不会信息量过载

### 8. AI情绪识别触发
**决策**: 用户点击按钮触发
**原因**: 节省API调用成本，用户可控

### 9. 情绪强度选择
**决策**: 滑块选择，分数保留两位小数
**原因**: 交互流畅，精度足够

### 10. 响应式
**决策**: 全站支持PC/移动端自适应
**原因**: 用户可能在手机上使用，需要良好体验
